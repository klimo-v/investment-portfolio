# CLAUDE.md

Инструкции для AI-ассистента (и для самого разработчика) по этому проекту.
Проект — приложение учёта инвестиционных сделок и портфелей. Стек и планы — в
`docs/` и `README.md`. Этот файл описывает **как писать код**: стандарты, архитектуру,
best practices.

Дополнительная цель проекта: он служит **тренировочным полигоном под собеседования**
на senior/архитектурные Angular-позиции. Поэтому разделы ниже намеренно затрагивают
темы из требований вакансий (SOLID, GRASP, FSD, RxJS, signals, безопасность, SSR,
state-менеджеры, производительность, CI/CD). См. §12 «Карта под собеседование».

## 2. Структура репозитория (Nx + FSD)

Комбинируем Nx-workspace с **Feature-Sliced Design (FSD)** внутри Angular-приложения —
это прямо из требований вакансий (FSD, feature isolation, lazy boundaries).

```
apps/
  web/                      # Angular 22 SPA
    src/app/
      app.config.ts         # provideRouter, provideHttpClient, zoneless
      app.routes.ts         # только lazy-загрузка фич
      pages/ (или app/)     # роут-уровень (FSD: app + pages)
  api/                      # NestJS
    src/
      modules/
        operations/ portfolios/ trades/ import/ quotes/
libs/
  core/                     # ДВИЖОК РАСЧЁТОВ + Zod-схемы (общий фронт/бэк)
  web-shared/               # UI-kit, темы, общие компоненты (styleguide)
```

**FSD-слои внутри `apps/web`** (сверху вниз, зависимости только вниз):

```
app        → инициализация, провайдеры, роутинг
pages      → страницы-роуты (Дашборд, Портфель, Операции, Импорт)
widgets    → крупные самостоятельные блоки (таблица позиций, форма операции)
features   → пользовательские сценарии (добавить операцию, импортировать файл)
entities   → бизнес-сущности (operation, position, trade, portfolio, instrument)
shared     → UI-kit, утилиты, типы, api-клиенты — ни от чего не зависит
```

Правило зависимостей FSD: слой может импортировать **только нижележащие**. Nx-теги
(`@nx/enforce-module-boundaries`) настраиваются так, чтобы нарушение границ ловил
линтер. Это и есть «feature isolation + lazy boundaries» из вакансии.

## 3. Angular — обязательные правила (из официального гайда v22)

Эти правила — не пожелания, а стандарт. AI-ассистент и разработчик им следуют всегда.

**Компоненты**
- Только **standalone** (не писать `standalone: true` — это дефолт). NgModules не используем.
- `input()` / `output()`-функции вместо декораторов `@Input` / `@Output`.
- `inject()` вместо инъекции через конструктор.
- `ChangeDetectionStrategy.OnPush` на всех компонентах.
- Малые, однозадачные компоненты (SRP). Инлайновый шаблон для маленьких.
- Host-биндинги — в объекте `host` декоратора, **не** через `@HostBinding`/`@HostListener`.

**Шаблоны**
- Нативный control flow **`@if` / `@for` / `@switch`** (не `*ngIf`/`*ngFor`/`*ngSwitch`).
- `class`/`style`-биндинги вместо `ngClass`/`ngStyle`.
- `@for` всегда с `track`.
- `NgOptimizedImage` для статичных изображений.
- Минимум логики в шаблоне — вычисления в `computed()`.

**Состояние**
- **Signals** для локального состояния; **`computed()`** для производных (итоги, P&L).
- Не использовать `mutate` на сигналах — только `set` / `update`.
- Чистые, предсказуемые трансформации состояния.

**TypeScript**
- Строгий режим. Избегать `any` — использовать `unknown` при неопределённости.
- Предпочитать вывод типов, когда тип очевиден.

**Формы**
- **Signal Forms** (стабильны в v22) для форм ввода операций — типобезопасно,
  декларативно, без ControlValueAccessor. Reactive Forms — если нужен legacy-паттерн.

**Данные**
- Загрузка через **`httpResource`** (реактивная обёртка над HttpClient на сигналах) с
  **`parse: zodSchema.parse`** для валидации ответа. Мутации (POST/PUT) — напрямую
  через `HttpClient`, не через `httpResource`.

## 4. Zoneless и change detection (тема собеса)

Проект — **zoneless** (Angular 22 по умолчанию, без Zone.js). Что это значит на практике:

- Change detection запускается **сигналами**, а не «дёрганьем зоны» на каждое событие.
  Меньше лишних проверок → выше производительность (тема «узкие места
  производительности» из вакансии).
- `OnPush` + сигналы — базовая связка. Компонент перерисовывается, когда изменился
  сигнал, который он читает, а не когда «что-то где-то произошло».
- Асинхронщина (setTimeout, промисы, RxJS) больше не триггерит CD автоматически —
  результат заводим в сигнал (через `toSignal`, `httpResource`, `resource`), и CD
  срабатывает от сигнала.

**Формулировка для собеса (signals vs zone, когда что):** Zone.js патчил все async
API, чтобы после каждого события прогонять глобальный CD — просто, но неэффективно и
непредсказуемо. Signals дают точечную, «pull»-реактивность: фреймворк знает граф
зависимостей и обновляет только затронутое. В новом коде — signals; Zone нужен лишь
для совместимости со старыми библиотеками, завязанными на автоматический CD.

## 5. RxJS — где ещё уместен и как правильно (тема собеса)

В signal-first проекте RxJS не исчезает — он остаётся для **потоков событий** (ввод,
WebSocket, дебаунс, гонки запросов). Мост между RxJS и сигналами: `toSignal` /
`toObservable`.

Правила и типовые формулировки для собеса:

- **Выбор flattening-оператора:**
  - `switchMap` — когда важен только последний запрос (поиск/автокомплит цен,
    смена выбранного портфеля): предыдущий запрос отменяется. **Осторожно с
    не-идемпотентными операциями** (POST) — отменённый запрос мог долететь до сервера.
  - `mergeMap` — параллельные независимые запросы без отмены.
  - `concatMap` — строгий порядок, по одному (последовательные записи).
  - `exhaustMap` — игнорировать новые, пока текущий не завершился (защита от
    дабл-клика по «Сохранить»/«Импортировать»).
- **Отписки:** `takeUntilDestroyed()` (Angular 16+) — предпочтительно; либо `async`
  pipe в шаблоне (сам отписывается). Ручные `subscribe` без отписки — источник утечек.
- **`shareReplay`:** для шаринга «горячего» результата между подписчиками; указывать
  `refCount: true` / `{ bufferSize: 1, refCount: true }`, иначе поток не завершится и
  будет держать подписку (утечка).
- **Race conditions:** решаются выбором правильного flattening-оператора (`switchMap`
  отменяет устаревшее) и отменой запросов (`httpResource` сам отменяет предыдущий при
  смене зависимости).

## 6. State-менеджмент: когда store, когда сервис+сигнал (тема собеса)

Дерево решений (прямо отвечает на «когда store оправдан»):

1. **Локальное состояние компонента** → `signal` / `computed` внутри компонента.
2. **Общее состояние фичи** → **сервис с сигналами** (`providedIn: 'root'` или на
   уровне роута), приватный `signal`, наружу — `readonly` через `computed`/`asReadonly`.
   Для большинства кейсов этого проекта — достаточно.
3. **Сложное глобальное состояние** (много источников, недо/redo, оптимистичные
   апдейты, серьёзная отладка) → **NgRx (SignalStore)** или **Akita**. Вводить только
   когда сервис-с-сигналом реально перестал справляться (YAGNI).

**Формулировка для собеса:** сервис+`Subject`/сигнал закрывает 80% задач; глобальный
store оправдан, когда нужны предсказуемость через единый редьюсер, инструменты
отладки (time-travel), нормализация и переиспользование состояния между несвязанными
частями приложения. NgRx **SignalStore** — современный мост: DX стора + реактивность
сигналов. Преждевременный NgRx — типичный оверинжиниринг (нарушение KISS/YAGNI).

В этом проекте: старт на **сервисах с сигналами**; NgRx SignalStore держим в уме на
случай роста (см. `docs/04-roadmap.md`).

## 7. Принципы проектирования (SOLID / GRASP / GoF) на примерах проекта

Разработка ведётся с осознанным применением принципов (это буквально пункты вакансий).

**SOLID**
- **S (SRP):** движок расчётов, парсеры брокеров, коннекторы котировок — разные
  модули с одной ответственностью. Компонент не считает P&L сам — берёт из `core`.
- **O (OCP):** новый брокер = новый парсер, реализующий общий интерфейс; ядро импорта
  не меняется. Новый источник котировок = новый провайдер.
- **L (LSP):** любой `BrokerParser` взаимозаменяем в конвейере импорта.
- **I (ISP):** узкие интерфейсы (`PriceProvider`, `BrokerParser`), а не «толстые».
- **D (DIP):** сервисы зависят от абстракций (токены/интерфейсы), реализации
  подставляются через DI — и в Angular, и в NestJS.

**GRASP** (полезно проговорить на собесе)
- **Information Expert:** расчёт средней цены живёт там, где данные позиции.
- **Creator / Factory:** создание парсера под брокера — через фабрику.
- **Controller:** NestJS-контроллер как точка входа сценария, логика — в сервисах.
- **Low Coupling / High Cohesion:** FSD-слои и Nx-границы обеспечивают низкую связность.
- **Polymorphism:** классификация операций и парсеры — через полиморфизм, не `switch`.

**GoF-паттерны, уместные здесь**
- **Strategy** — метод учёта (средняя / FIFO), стратегия импорта под брокера.
- **Adapter** — коннекторы MOEX/ЦБ/Binance к единому `PriceProvider`.
- **Facade** — сервис-фасад фичи для страницы, скрывающий детали.
- **Factory Method** — выбор парсера по типу файла/брокера.

**DRY / KISS / YAGNI** — общие суммы и схемы в `libs/core` (DRY); монолит вместо
микросервисов на старте (KISS); NgRx/SSR/микрофронты только когда понадобятся (YAGNI).

**Антипаттерны, которых избегаем:** God Component, бизнес-логика в шаблоне, ручные
подписки без отписки, `any`, дублирование расчётов на фронте и бэке, преждевременная
оптимизация.

## 8. Безопасность фронтенда (XSS, CSRF — тема вакансий)

- **XSS:** доверяем встроенной санитизации Angular; **не** обходим её через
  `bypassSecurityTrust*` без крайней необходимости; никакого `innerHTML` с
  непроверенными данными. CSP-заголовки на бэке.
- **CSRF/XSRF:** используем `withXsrfConfiguration` в `provideHttpClient`; на бэке —
  проверка токена. Для API — куки `SameSite`, `HttpOnly`.
- **Валидация на входе:** Zod и на фронте (формы), и на бэке (тело запроса) —
  доверять данным клиента нельзя.
- **Секреты** (ключи API брокеров) — только на бэке, никогда в бандл фронта.
- **Правила безопасной разработки:** зависимости через аудит (`pnpm audit`),
  принцип наименьших привилегий, отсутствие чувствительных данных в логах.

## 9. Производительность и «узкие места» (тема вакансий)

- **OnPush + сигналы + zoneless** — базовая производительность из коробки.
- **Ленивая загрузка** роутов-фич (`loadComponent`/`loadChildren`) — маленький
  начальный бандл.
- `@for` с `track` — минимальные перерисовки больших таблиц позиций.
- `@defer`-блоки — отложенная загрузка тяжёлых виджетов (графики, большие таблицы).
- Виртуальный скролл (CDK Virtual Scroll) для длинных списков операций.
- Профилирование: Angular DevTools (профайлер CD), source-map-explorer для бандла.
- Мемоизация тяжёлых расчётов в `computed()` (пересчёт только при смене зависимостей).

## 10. Тестирование (тема вакансий: авто-тесты)

- **Vitest** — юнит-тесты. Приоритет №1 — **движок расчётов в `libs/core`**:
  средняя цена, P&L, реализованный/нереализованный, частичные продажи — на **ваших
  реальных цифрах** из `docs/02-data-model.md`.
- Тесты парсеров импорта на примерах реальных отчётов брокеров.
- Компонентные тесты Angular через TestBed; `httpResource`/HttpClient — через
  `provideHttpClientTesting`.
- E2E (позже) — Playwright для ключевых сценариев (добавить операцию, импорт).
- Цель — не 100% покрытие, а покрытие **финансовой логики**, где ошибка дороже всего.

## 11. Рабочий процесс для AI-ассистента

- Перед изменением Angular-кода свериться с официальным гайдом (Angular CLI MCP:
  `get_best_practices`, `search_documentation`) — стандарты обновляются.
- Не хранить деньги во `float`/`number` — decimal/целые единицы (см. `docs/01`, §4).
- Не вводить новые крупные зависимости (NgRx, SSR, микрофронты) без явного запроса —
  YAGNI. Сначала предложить и объяснить trade-off.
- Соблюдать FSD-границы и Nx-теги; не тянуть импорт «вверх» по слоям.
- Любую правку финансовых формул сопровождать/обновлять тестом.
- Коммиты — атомарные, с осмысленным сообщением; ветки под фичи.
- Идентификаторы кода — на английском; комментарии и документация — на русском (как в проекте).
