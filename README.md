# Investment Portfolio — учёт сделок и портфелей

Личное приложение для учёта инвестиционных сделок по нескольким торговым системам
и брокерам. Заменяет ручное ведение в Google Sheets. На старте — локальный веб на
своём ПК, с заложенной возможностью расширения до десктоп-приложения и публичного
SaaS без переписывания кода.

## Проблема, которую решаем

Сейчас сделки из разных брокеров вносятся вручную в три листа Google Sheets
(Сделки, Закрытые сделки, Портфель), причём два из них дублируются руками. Это
долго и приводит к ошибкам. Цель — вводить/импортировать данные **в одном месте**,
а позиции, сделки и итоги считать **автоматически**.

## Стек (кратко)

TypeScript везде. Фронтенд — **Angular 22** (standalone, signal-first, zoneless) +
Angular Material + Signal Forms. Бэкенд — **NestJS** (Fastify-адаптер) + Drizzle ORM
+ SQLite (→ PostgreSQL в SaaS). Валидация — Zod. Монорепо — Nx. Десктоп позже —
Tauri 2. Подробности в `docs/01-tech-stack.md`.

## Ключевые решения

- **Один источник истины** — журнал операций; позиции и сделки вычисляются, не
  дублируются руками.
- **Деньги — не во float** — decimal/целые единицы, чтобы не терять копейки.
- **Импорт как плагины** — по одному парсеру на брокера, общее ядро.
- **Рыночные данные по API** — MOEX ISS, ЦБ РФ, крипто-API.
- **Тесты движка расчётов на реальных цифрах** — защита от ошибок в формулах.

## Глоссарий

- **Система** — торговая стратегия (Верников, Верников_трейдинг, Трейдинг_крипта).
- **Портфель** — брокерский счёт / назначение (позволяет вести переводы между ними).
- **Операция** — одна строка журнала (покупка, продажа, дивиденд, налог, перевод…).
- **Сделка (Trade)** — группа операций по инструменту; бывает открытой/закрытой.
- **Позиция** — текущий остаток по инструменту в портфеле.

## Структура репозитория

Nx-монорепо (см. `CLAUDE.md` §2):

```
apps/
  web/      Angular 22 — standalone, zoneless, FSD-слои, lazy-роуты, Material
  api/      NestJS (Fastify) — модуль operations использует общий движок @core
libs/
  core/     движок расчётов (decimal.js) + Zod-схемы — общий фронт/бэк
  web-shared/  UI-утилиты (форматирование денег, P&L-цвета)
```

## Запуск

Требуется Node 22+ и pnpm.

```bash
pnpm install          # установить зависимости
pnpm start            # Angular фронт → http://localhost:4200
pnpm start:api        # NestJS бэк → http://localhost:3000/api
pnpm test             # тесты (движок расчётов в libs/core)
pnpm lint             # проверка + Nx-границы FSD
pnpm graph            # граф зависимостей проектов

# база данных (Drizzle + SQLite)
pnpm db:generate      # сгенерировать SQL-миграцию из apps/api/src/db/schema.ts
pnpm db:migrate       # применить миграцию к data/portfolio.sqlite
pnpm db:seed          # (опционально) справочники: системы/портфели/инструменты
pnpm db:studio        # Drizzle Studio — просмотр БД в браузере
```

Если на вашей машине pnpm ругается на симлинки в HOME — раскомментируйте
`node-linker=hoisted` в `.npmrc`.

## Статус

**Фаза 0 и Фаза 1 завершены** (`docs/04-roadmap.md`): Nx-монорепо, Angular 22 SPA
с FSD и 5 страницами, NestJS-бэк на Drizzle + SQLite (не in-memory), общий движок
расчётов в `libs/core`. **Движок покрыт тестами на реальных цифрах — все проходят**
(SBER: вложено 497 783.03, P&L 9 214.09; частичные продажи; дивиденды; переводы
между портфелями не искажают P&L).

API: `OperationsModule` (CRUD операций + расчёт позиций через `@core`) и
`PortfoliosModule` (справочники систем/портфелей/инструментов) — оба читают/пишут
в SQLite через Drizzle.

Дальше — Фаза 2 (`docs/04-roadmap.md`): страницы Операции/Портфель/Сделки на
Signal Forms и реальных `httpResource`-запросах к API вместо демо-данных.

> `better-sqlite3`) — вы уже запустили `pnpm install`/`nx serve web` локально, и
> это сработало. Изолированно проверено: движок `libs/core` (6/6 тестов) и
