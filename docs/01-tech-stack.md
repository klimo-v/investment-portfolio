# 01 — Стек технологий (2026)

Документ описывает, на чём разрабатывать приложение для учёта сделок и портфелей.
Ключевое ограничение из требований: **сначала веб на своём ПК, с возможностью
позже «перегнать» в десктоп и в публичный SaaS без переписывания**.

Стек подобран под ваш профиль: вы **фронтенд-разработчик на Angular**, поэтому фронт —
**Angular 22**, а бэкенд — **NestJS**, который сознательно повторяет архитектуру
Angular (модули, DI через декораторы, `inject()`). Один язык (TypeScript), одна
ментальная модель на фронте и на бэке.

---

## 1. Итоговая рекомендация (коротко)

| Слой | Выбор | Почему |
|---|---|---|
| Язык | **TypeScript** (фронт и бэк) | Один язык на весь проект, строгая типизация — критично для финансов |
| Фронтенд | **Angular 22** (standalone, signal-first, zoneless) | Ваш основной инструмент; в v22 — signal-first и zoneless по умолчанию |
| Состояние | **Signals** (`signal`, `computed`, `resource`) | Встроено в Angular 22, не нужен внешний стор для большинства кейсов |
| Формы | **Signal Forms** (стабильны в v22) | Типобезопасные декларативные формы без ControlValueAccessor |
| HTTP/серверное состояние | **`HttpClient` + `resource()`/`httpResource`** | Реактивная загрузка данных на сигналах, встроенный кэш-паттерн |
| Роутинг | **Angular Router** (lazy-loaded routes) | Встроен, ленивая загрузка фич из коробки |
| UI-компоненты | **Angular Material 22 + CDK** | Родной для Angular, доступный, есть таблицы/датапикеры/диалоги |
| Таблицы | **Material Table + CDK** (или **AG Grid** для тяжёлых) | Сортировка, фильтры, группировка «как в Sheets» |
| Графики | **ngx-echarts** (ECharts) или **ng2-charts** (Chart.js) | Динамика P&L, доли портфеля, свечи |
| Бэкенд | **NestJS** (на Node.js) | Ангуляр-подобная архитектура: модули, DI, декораторы — знакомо вам |
| Веб-слой Nest | **Fastify-адаптер** для NestJS | Nest поверх быстрого Fastify (а не Express) |
| ORM / БД-доступ | **Drizzle ORM** (или **Prisma**) | Типобезопасные запросы, миграции; Drizzle легче, Prisma привычнее многим |
| База данных | **SQLite (локально) → PostgreSQL (SaaS)** | Один ORM; на старте файл на диске, потом облачная БД |
| Валидация | **Zod** (+ `nestjs-zod`) | Общие схемы для фронта, бэка и разбора импорта |
| Сборка/тесты | **Vitest** (дефолт в Angular 22 и Nest) | Единый тест-раннер на фронте и бэке |
| Десктоп-обёртка | **Tauri 2** | ~5 МБ вместо ~100 МБ у Electron; внутри — тот же Angular-билд |
| Рыночные данные | **MOEX ISS API, ЦБ РФ, крипто-API** | Автоподтяжка цен и курсов (см. §5) |
| Импорт брокеров | Свой **модуль-парсер** в NestJS (Excel/CSV → единый формат) | См. §6 |

> Одной фразой: **Angular 22 на фронте + NestJS на бэке, оба на TypeScript и оба с
> DI-архитектурой. Drizzle+SQLite сейчас → Postgres-SaaS и Tauri-десктоп потом,
> без переписывания.**

## 2. Главное архитектурное решение: web → desktop → SaaS

Вопрос «веб или десктоп» решается не выбором «или-или», а **разделением на два слоя**:

```
┌─────────────────────────────────────────────┐
│  ФРОНТЕНД (Angular 22, standalone SPA)        │  ← пишется ОДИН раз
│  таблицы, Signal Forms, графики, весь UI      │
└───────────────┬─────────────────────────────┘
                │ HTTP / JSON REST API
┌───────────────┴─────────────────────────────┐
│  БЭКЕНД (NestJS + Fastify + Drizzle)          │  ← пишется ОДИН раз
│  расчёты, импорт, доступ к БД, рыночные цены  │
└───────────────┬─────────────────────────────┘
                │
        ┌───────┴────────┐
     SQLite            PostgreSQL
   (локально)          (SaaS позже)
```

Один и тот же фронтенд запускается в трёх режимах:

- **Сейчас (веб на своём ПК):** `ng serve` поднимает фронт, NestJS — локально. БД —
  файл SQLite. Никакого интернета, кроме подтяжки цен.
- **Позже (десктоп):** production-билд Angular упаковывается в **Tauri** — обычное
  приложение с иконкой, офлайн, БД в папке приложения. Код фронта не меняется.
- **В будущем (SaaS):** тот же билд разворачивается на сервере, NestJS переключается
  на PostgreSQL, добавляются авторизация и мультипользовательность. Drizzle позволяет
  сменить SQLite→Postgres малой кровью.

**Вывод по вашему вопросу «веб или десктоп»:** начинаем с веба (быстрее итерации),
но фронт и бэк раздельно с самого начала → десктоп через Tauri добавляется за 1–2 дня.

## 3. Почему именно этот стек (под Angular-разработчика)

**Angular 22 — ваш основной инструмент, и он сейчас в лучшей форме.** Версия от июня
2026 — «signal-first»: реактивность на сигналах встроена, **zoneless** по умолчанию
(без Zone.js — меньше магии, предсказуемее и быстрее change detection), **Signal
Forms** стабильны, **Vitest** — дефолтный тест-раннер. Это ровно то, что нужно для
приложения с таблицами и формами: состояние портфеля описывается сигналами,
производные величины (итоги, P&L) — через `computed()`, а формы ввода операций — на
Signal Forms.

**NestJS на бэке — потому что он «Angular для сервера».** Те же модули, тот же DI
через декораторы, тот же `inject()`-стиль. Вам не придётся осваивать чужую парадигму:
контроллеры/сервисы/провайдеры в Nest устроены как в Angular. Под капотом ставим
**Fastify-адаптер** — Nest работает поверх быстрого Fastify вместо Express.

**Drizzle + SQLite.** SQLite — просто файл: идеален для локального приложения,
бэкап копированием, ноль настройки. Drizzle даёт типобезопасные запросы и миграции.
Смена на PostgreSQL для SaaS — минимальная. (Если вам ближе Prisma — он тоже
отлично работает с Nest; Drizzle легче и ближе к SQL.)

**Zod везде.** Одна схема описывает валидацию Signal Forms на фронте, проверку тела
запроса в Nest (через `nestjs-zod`) и разбор импортируемого файла брокера. Для
финансов важно: битая строка отсекается на входе, а не ломает расчёты.

**Material вместо сторонних UI-китов.** Angular Material 22 + CDK — родной для
Angular, доступный, содержит готовые таблицы, датапикеры, диалоги, что закрывает
почти весь ваш UI (формы ввода, таблицы позиций). Для очень больших таблиц с
группировкой можно добавить AG Grid.

## 4. Точность денег: обязательное правило

Финансовые суммы **нельзя** хранить и считать в обычных числах с плавающей точкой
(`number`) — накапливается ошибка округления. Два подхода:

- Хранить в БД как **целые в минимальных единицах** (копейки/центы) или как `decimal`
  (в Postgres — тип `numeric`).
- В расчётах использовать **decimal-библиотеку** (`decimal.js`), а не арифметику
  JS-`number`.

Это заложено в модель данных (`02-data-model.md`) и особенно важно для мультивалютных
позиций (RUB/USD) и пересчёта по курсу. Движок расчётов живёт в общем коде и
переиспользуется фронтом и бэком.

## 5. Рыночные данные (по API, как вы выбрали)

| Что нужно | Источник | Примечание |
|---|---|---|
| Цены акций/облигаций/ETF ММВБ | **MOEX ISS API** | Бесплатный, официальный; цены, история, купоны. Основной источник |
| Курсы валют (USD/RUB и др.) | **API ЦБ РФ** (cbr.ru) | Официальный курс на дату — для пересчёта в RUB |
| Крипто (BTC, USDT) | **Binance / CoinGecko API** | Цена в USD, дальше в RUB через курс |
| Резерв по акциям РФ | T-Bank Invest API | Если понадобится — см. §6 |

Модуль «Котировки» в NestJS периодически (или по кнопке «Обновить») тянет цены и
курсы, кэширует их в БД и пересчитывает текущую стоимость. На фронте это отражается
через `resource()`/`httpResource` — данные реактивно обновляются в сигналах. Ручная
правка цены остаётся как fallback.

## 6. Импорт сделок от брокеров

Ваш подход правильный и заложен в архитектуру: **выгрузка из брокера за период →
файл → парсер → единый формат → движок сам определяет тип операции.**

Конвейер импорта (модуль `import` в NestJS):

```
Файл брокера (.xlsx / .csv / .pdf)
        │
        ▼
[1] Парсер под конкретного брокера (Tinkoff, БКС, …)
        │   приводит строки к единому виду
        ▼
[2] Нормализация → единая схема сделки (Zod-валидация)
        │
        ▼
[3] Движок классификации:
        • перевод между портфелями (Between-portfolio)
        • покупка / продажа
        • дивиденд / купон / налог / комиссия
        • депозит / вывод кэша
        │
        ▼
[4] Дедупликация (чтобы не задвоить при повторной загрузке)
        │
        ▼
[5] Сопоставление в позиции (средняя/FIFO) → пересчёт портфеля
```

Каждый брокер = отдельный провайдер-парсер, реализующий общий интерфейс (в стиле
Nest-провайдеров). Добавить брокера = написать один парсер, остальное переиспользуется.

**Официальные SDK для автоматической выгрузки (2-я фаза):** у T-Bank (Tinkoff) есть
Invest API. На старте это не нужно — начинаем с файлов, как вы и планировали.

## 7. Инструменты разработки

| Назначение | Инструмент |
|---|---|
| CLI/скаффолдинг | **Angular CLI** (`ng`) и **Nest CLI** (`nest`) |
| Тесты | **Vitest** — дефолт в Angular 22 и в Nest; покрыть движок P&L и импорт |
| E2E-тесты (позже) | Playwright |
| Пакетный менеджер | **pnpm** (быстрый, экономит место) |
| Монорепо | **Nx** — родной для Angular/Nest, удобно держать фронт+бэк+общий код вместе |
| Линт/формат | ESLint (angular-eslint) + Prettier |
| Версионирование | Git (репозиторий уже инициализирован) |

**Nx** особенно уместен: он изначально из мира Angular, умеет один репозиторий с
`apps/web` (Angular), `apps/api` (Nest) и `libs/core` (общий движок расчётов и
Zod-схемы), с кэшированием сборок и генераторами.

Тесты на движок расчётов — не «излишество», а страховка: любая правка формулы P&L
прогоняется на ваших реальных цифрах из примеров.

## 8. Официальные правила Angular, которых держимся (v22)

Из официального гайда Angular — чтобы код был современным и не устаревшим:

- **Standalone-компоненты** (без NgModules); `standalone: true` не пишем — это дефолт.
- **Signals для состояния**, `computed()` — для производных величин (итоги, P&L).
- **`input()` / `output()`-функции** вместо декораторов `@Input`/`@Output`.
- **`inject()`** вместо инъекции через конструктор.
- **Нативный control flow** `@if` / `@for` / `@switch` вместо `*ngIf`/`*ngFor`.
- **`ChangeDetectionStrategy.OnPush`** на компонентах (в v22 — по сути дефолт-путь).
- **Ленивая загрузка** маршрутов фич (Операции, Портфель, Импорт грузятся по отдельности).
- Host-биндинги — в объекте `host` декоратора, не через `@HostBinding`/`@HostListener`.
- **`class`/`style`-биндинги** вместо `ngClass`/`ngStyle`.
- Строгий TypeScript, без `any` (использовать `unknown` при неопределённости).

## 9. Чего избегать на старте

- **Не начинать с микросервисов / Kubernetes / Docker-оркестрации.** Для локального
  MVP монолит-NestJS + SQLite достаточно.
- **Не тащить внешний стор (NgRx и т.п.) раньше времени.** В Angular 22 сигналов
  хватает для состояния портфеля; NgRx — только если состояние реально разрастётся.
- **Не хранить деньги во `number`.** См. §4.
- **Не хардкодить логику одного брокера в ядро.** Парсеры — провайдеры (§6).
- **Не откладывать тесты расчётов.** Финансовая ошибка дороже времени на тест.
