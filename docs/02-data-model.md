# 02 — Модель данных и архитектура

Здесь описано, **как хранить данные**, чтобы три ваших листа (Сделки, Закрытые
сделки, Портфель) перестали быть тремя ручными таблицами и стали одной базой,
из которой всё остальное считается автоматически.

---

## 1. Главная идея: одна таблица истины

Сейчас у вас три листа, и два из них (Закрытые сделки, Портфель) вы фактически
поддерживаете руками, дублируя данные. Это источник ошибок.

**Правильно так:** есть **один** источник истины — журнал операций (лист «Сделки»).
Всё остальное — **вычисляемые представления**, которые программа строит сама:

```
                    ┌────────────────────┐
                    │  ОПЕРАЦИИ (журнал)  │   ← единственное, что вы вводите/импортируете
                    │  = ваш лист «Сделки» │
                    └──────────┬─────────┘
                               │  движок расчётов (пересчёт при любом изменении)
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
      ┌─────────────┐  ┌──────────────┐  ┌──────────────┐
      │  ПОЗИЦИИ     │  │  СДЕЛКИ      │  │  ПОРТФЕЛИ    │
      │ (открытые)   │  │ (закрытые)   │  │  (сводка)    │
      │ = «Портфель» │  │ = «Закрытые» │  │  = итоги     │
      └─────────────┘  └──────────────┘  └──────────────┘
        вычисляется       вычисляется        вычисляется
```

Вы больше **не заполняете** «Портфель» и «Закрытые сделки» руками — они всегда
пересчитываются из журнала операций. Ввод только в одном месте.

## 2. Сущности (таблицы БД)

### 2.1. `operations` — журнал операций (ядро)

Это ваш лист «Сделки», приведённый к строгой схеме. Одна строка = одна операция.

| Поле | Тип | Описание | Из вашей таблицы |
|---|---|---|---|
| `id` | uuid | Внутренний ID | — |
| `date` | date | Дата операции | Дата_формат |
| `system_id` | fk → systems | Торговая система | Система |
| `portfolio_id` | fk → portfolios | Портфель/брокерский счёт | Брокер + Система |
| `instrument_id` | fk → instruments | Инструмент (null для кэш-операций) | Тикер + Тип инструмента |
| `operation_type` | enum | Тип операции (см. §3) | Тип сделки |
| `quantity` | decimal | Количество | Количество |
| `price` | decimal | Цена за единицу | Цена |
| `amount` | decimal | Сумма (в валюте сделки) | Сумма |
| `fee` | decimal | Комиссия | Комиссия |
| `leverage` | text | Плечо (1x, 2x…) | Плечо |
| `fx_rate` | decimal | Курс валюты к RUB на дату | Курс к RUB |
| `total_ccy` | decimal | Итого в валюте сделки | Итого (в валюте сделки) |
| `total_rub` | decimal | Итого в RUB | Итого RUB |

## 3. Типы операций (enum `operation_type`)

Собрано из ваших данных — движок должен уметь распознавать все эти виды:

| Тип | Что делает с позицией | Пример из ваших данных |
|---|---|---|
| `Deposit` | Ввод кэша на счёт | Deposit RUB 497 782 |
| `Withdraw` | Вывод кэша | Withdraw RUB 505 799 |
| `Buy` | Покупка → открывает/увеличивает позицию | Buy SBER 1600 |
| `Sell` | Продажа → уменьшает/закрывает позицию | Sell SBER 1600 |
| `Dividend` | Дивиденд → в `dividends_rub` | LKOH 31 733 |
| `Coupon` | Купон по облигации → в `coupons_rub` | RU000A10B4K3 21 462 |
| `Tax` | Налог → уменьшает результат | Tax RUB 1 198 |
| `Fee` | Отдельная комиссия | — |
| `Transfer` | Перевод между портфелями (парой строк) | Между портфелями = Да |

**Перевод между портфелями** — важный кейс: это две связанные операции (списание из
портфеля А + зачисление в портфель Б), объединённые `transfer_group`. Движок не
должен считать это покупкой/продажей и не должен искажать P&L.

## 4. Движок расчётов (что программа считает сама)

Логика, которую вы сейчас держите в формулах Google Sheets, переносится в код и
покрывается тестами.

**Метод учёта:** по умолчанию **средневзвешенная цена** (как у вас сейчас — «Средняя
цена покупки»). Опционально можно добавить FIFO — это влияет на расчёт налога и
реализованного P&L. Метод задаётся в настройках.

Что пересчитывается при любом изменении журнала:

1. **Средняя цена покупки** позиции считается **по оставшимся (не проданным) лотам**,
   а не по всем историческим покупкам. Важно: в ваших текущих данных «Средняя цена»
   и «Вложено» не всегда связаны как `вложено = кол-во × средняя` — например, у SBER
   (400 шт, средняя 303.51) «Вложено» 76 742.95 — это **нетто после частичной
   продажи** 240 шт, а не 400 × 303.51. Движок должен различать: средняя цена — по
   методу учёта (средневзвешенная/FIFO) на остаток; «Вложено» — фактически вложенный
   капитал с учётом реализованных частей. Оба показателя тестируются на ваших цифрах.
2. **Вложено** (в валюте и в RUB, с комиссией) — с корректной обработкой частичных
   продаж.
3. **Реализованный P&L** — при продажах (выручка − себестоимость проданной части).
4. **Нереализованный P&L** — по текущей котировке (текущая стоимость − вложено).
5. **Дивиденды/купоны** — накапливаются в позицию и в сделку.
6. **Пересчёт в RUB** — по `fx_rate` на дату операции (исторический) и по текущему
   курсу (для текущей стоимости).
7. **Статус сделки** — открыта / частично / закрыта по остатку количества.
8. **Сводка по системе и по портфелю** — агрегаты для дашборда.

> Ключевое: вы вводите операцию → всё остальное (позиции, сделки, итоги)
> пересчитывается автоматически. Ручного дублирования нет.

## 5. Мультивалютность (RUB / USD / крипто)

В ваших данных есть RUB, USD (облигация `RU000A10B4K3`, крипта). Правила:

- У каждой операции — **валюта сделки** и **курс к RUB на дату** (исторический,
  фиксируется в момент операции — не пересчитывается задним числом).
- Текущая стоимость считается **и в валюте инструмента, и в RUB** по **текущему**
  курсу из `quotes`.
- Итоги портфеля показываются в RUB (базовая валюта отчётности), с возможностью
  переключить на валюту инструмента.

---

## 6. Дедупликация при импорте

Чтобы повторная загрузка отчёта не задвоила операции:

- Каждая операция получает `broker_ref` (номер операции у брокера) — если он есть.
- При отсутствии реф-номера — хэш из (дата + тикер + тип + количество + сумма).
- Импорт помечается `import_batch_id`, любую загрузку можно откатить целиком.
