### Фаза 3.1 — Разнородный отчёт: несколько систем и ИИС/брокерский счёт

Цель: один отчёт брокера может содержать операции **разных торговых систем** и лежащие
на **разных счетах** (обычный брокерский vs ИИС). Значит батч-разметка «один портфель +
одна система на весь файл» недостаточна — нужна построчная/по-правилам разметка.

**Ключевой принцип: две ортогональные оси — два способа разметки.**

- **Счёт / портфель (брокерский ↔ ИИС) — объективно, часто есть в отчёте.**
  ИИС и брокерский — это *разные записи* в `portfolios` под одним брокером
  (`docs/02-data-model.md §2.3`). Парсер извлекает признак счёта на каждую строку
  (торговый код / пометка «ИИС» / раздел-площадка отчёта), а маппинг
  «признак счёта в отчёте → `portfolio_id`» задаётся один раз и запоминается.
  Батч-выбор портфеля остаётся как fallback, когда признак не распознан.
  *(Опционально: поле `portfolios.account_type ∈ {Brokerage, IIS}` для налоговой
  отчётности — не обязательно для работы импорта, YAGNI.)*

- **Торговая система — субъективная разметка пользователя, в отчёте её нет НИКОГДА,
  и, в отличие от счёта, НЕ persisted-маппинг.** Один и тот же тикер в разное время
  может относиться к разным системам (это решение пользователя в моменте, а не
  свойство тикера) — поэтому запоминать правило «тикер → система» между импортами
  неправильно. Решается точечным выбором системы по тикеру **в рамках текущего
  импорта** (`tickerSystemOverrides`, живёт только в текущем запросе/сессии
  страницы); нераспознанное помечается `warn` («Проверить») и спрашивается заново
  при каждой загрузке.

**Шаги (инкременты):**

- [x] **Редактирование операций** (`system_id`, `portfolio_id`) у уже сохранённых —
      фундамент. `PATCH /operations/:id` (Zod `OperationReassignSchema`, хотя бы одно
      поле), массовый выбор чекбоксами на странице Операции (переиспользуют чекбоксы
      удаления) → диалог «Назначить систему/портфель»; колонки Система/Портфель видны
      в таблице.
- [x] **Признак счёта в парсере**: `RawRow.accountRef`/`accountKind` ('IIS' |
      'Brokerage'), заполняются в `broker-html-report.ts` последовательным обходом
      документа (таблицы + текст между ними по порядку) — маркер счёта из текста
      («Договор на ведение ИИС…», «Торговый код: …») действует на все таблицы после
      него, до следующего маркера. Резолвер портфеля (`import.service.ts`) пробует
      accountRef (по имени портфеля) прежде батч-дефолта. Тесты: реальный отчёт
      (один счёт ИИС SESPS) + синтетический документ с двумя счетами (механизм
      переключения контекста).
- [x] **Маппинг «счёт отчёта → портфель»** (persisted): поле `portfolios.account_ref`
      (nullable, unique; миграция `0002_common_excalibur.sql`). Авто-обучение при
      `commit()` — если у операции есть `accountRef`, а он ещё не привязан ни к
      одному портфелю, привязывается к портфелю, куда фактически попала операция
      (батч-дефолт или уже отработавший резолвер), и не перезаписывается молча ни
      при конфликте (accountRef уже привязан к другому портфелю), ни если у целевого
      портфеля уже есть свой accountRef. Дальнейшие импорты того же счёта резолвятся
      автоматически, батч-выбор портфеля становится не нужен. Показывается read-only
      в диалоге «Портфели». Проверено end-to-end на живом API (learn → приоритет над
      батч-дефолтом → защита от перезаписи), тестовые данные полностью откачены.
- [x] **Выбор системы по тикеру — НЕ persisted** (сознательная поправка после
      обратной связи: «для одного тикера может быть несколько систем, запоминать
      не имеет смысла, нужно каждый раз при импорте спрашивать»). Изначально была
      реализована persisted-таблица `system_rules` (миграция `0003_smart_snowbird`)
      по аналогии с accountRef→портфель — откатил полностью (миграция и таблица
      удалены, схема и БД сведены к состоянию без неё). Взамен: `ImportOptions.
      tickerSystemOverrides: Record<ticker, systemId>` — карта живёт только в
      текущем запросе `preview()`/`commit()` (на фронте — в сигнале страницы,
      сбрасывается при загрузке нового файла). `resolveSystem(name, ticker)`
      пробует override после явного имени, прежде батч-дефолта. Строка без явного
      выбора не блокируется (не `error`) — импортируется с батч-дефолтом, но
      помечается `warn` + `systemUncertain: true` c причиной («выберите систему
      для тикера»). UI: панель под сводкой с уникальными тикерами без явного
      выбора, `mat-select` применяется сразу (без отдельной кнопки «запомнить»).
      Тесты в `import.spec.ts` (явный выбор vs батч-дефолт vs резолвер без
      `systemChosenForTicker` — обратная совместимость; отдельный тест — один и
      тот же тикер в двух независимых вызовах уходит в разные системы). Проверено
      end-to-end на живом API: без override — warn/батч-дефолт, с override — ok/
      выбранная система, следующий вызов без override — снова warn (ничего не
      персистится).
- [ ] **Редактируемый предпросмотр по строкам**: точечная правка Системы/Портфеля
      прямо в конкретной строке предпросмотра (без создания персистентного правила) —
      сейчас разметка либо автоматическая (правила/accountRef), либо батч-дефолт;
      точечная ручная правка так и не понадобившихся строк делается уже после
      импорта через массовое назначение на странице Операции (готово, инкремент 1).
      Оставлено как возможное дальнейшее уточнение UX, не блокирует сценарий.

**Итог подфазы:** один разнородный отчёт (разные системы + ИИС/брокерский счета)
корректно раскладывается по портфелям и системам — портфель резолвится по
persisted-маппингу счёта, система — по persisted-правилам тикера; для нового
тикера/счёта система один раз спрашивает и запоминает, дальше действует сама.
Остаточная точечная правка (если правило не заводили) — через Операции → массовое
назначение.

## Что делаем первым (ответ на «с чего начать»)

**Фаза 0 → Фаза 1 → Фаза 2.** То есть: каркас → движок расчётов с тестами на ваших
цифрах → три основных экрана. Уже после Фазы 2 вы полностью уходите из Google Sheets
для ручного учёта. Импорт (Фаза 3) и автоцены (Фаза 4) — сразу следом, они закрывают
вашу главную боль (ручной ввод).

Оценки в днях — ориентир для одного человека с ассистентом-генератором кода;
реальный темп зависит от того, сколько времени вы уделяете проекту.
